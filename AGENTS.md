# AGENTS.md

## Текущее состояние проекта

Проект представляет собой Telegram-бот для ежедневного планирования, шаблонной подготовки идей роликов и запуска обучающего Mini App с простым backend и админ-панелью. Код бота находится в `bot.py`, backend — в `server.py` и `database.py`, единый запуск для деплоя — в `start.py`, интерфейс Mini App — в `webapp/`, зависимости — в `requirements.txt`, а инструкции — в `README.md`.

## Стек проекта

- Язык программирования: Python 3.10+.
- Telegram-фреймворк: `python-telegram-bot` 22.8.
- Web backend: `Flask` 3.1.2.
- Загрузка переменных окружения: `python-dotenv` 1.2.2.
- Хранилище данных курса: SQLite через стандартный модуль `sqlite3`; файл по умолчанию `course.db`.
- Хранилище дневных планов бота: словарь в памяти процесса; отдельной базы для планов нет.
- Менеджер зависимостей: `pip`, зависимости зафиксированы в `requirements.txt`.
- Деплой: Railway запускает backend и Telegram-бота одной командой `python start.py`.
- Автоматические тесты, линтер и форматтер пока не настроены.
- Отдельная сборка и контейнеризация не требуются.
- Mini App: HTML, CSS и JavaScript без сборщика; данные курса загружаются через API backend.

## Запуск и сборка

Все команды выполняются из корня проекта.

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
Copy-Item .env.example .env
python start.py
```

Синтаксическая самопроверка:

```powershell
python -m py_compile bot.py database.py server.py start.py
```

Локальный запуск всего проекта:

```powershell
python start.py
```

После запуска откройте `http://127.0.0.1:8000` для Mini App и `http://127.0.0.1:8000/admin` для админки. На Railway задайте `DATABASE_PATH=/data/course.db` и подключите Volume с mount path `/data`.

Отдельной команды сборки нет.

## Правила кода

- When writing complex features or significant refactors, use an ExecPlan (as described in PLANS.md) from design to implementation.
- Следуйте конфигурации форматтера и линтера, добавленной в репозиторий; не вводите параллельный стиль вручную.
- Делайте небольшие, тематически цельные изменения и не затрагивайте несвязанные файлы.
- Отделяйте обработчики Telegram от бизнес-логики и работы с внешними сервисами.
- Формирование контент-плана оставляйте в чистой функции `build_content_plan`, чтобы шаблоны можно было проверять без Telegram API.
- Не добавляйте реальные токены и адреса окружений в `.env.example`; публичный адрес Mini App берите из `WEBAPP_URL`.
- Данные модулей и уроков храните в SQLite через функции `database.py`; не возвращайте Mini App к захардкоженному массиву уроков.
- Админские endpoint’ы должны проверять пароль из `ADMIN_PASSWORD`.
- Не храните токены, пароли, ключи API и другие секреты в исходном коде. Используйте переменные окружения и добавьте файл-пример без реальных значений.
- Проверяйте и нормализуйте входные данные пользователя. Ошибки внешних API обрабатывайте явно и не показывайте пользователю внутренние трассировки.
- Для новой логики добавляйте тесты после появления тестовой инфраструктуры.
- Обновляйте README и этот файл при изменении команд, структуры проекта или обязательных инструментов.
- Не добавляйте зависимости без практической необходимости; фиксируйте их назначение в документации или описании изменения.

## Самопроверка

Перед завершением работы:

1. Просмотрите итоговые изменения и убедитесь, что в них нет секретов, временных файлов и отладочного вывода.
2. Выполните настроенные форматирование, линтинг, проверку типов и тесты. Пока соответствующих команд в проекте нет, явно сообщайте, что автоматические проверки недоступны.
3. Если менялись зависимости, проверьте установку в чистом окружении и актуальность lock-файла.
4. Если менялось поведение бота, вручную проверьте затронутый сценарий: успешный ответ, некорректный ввод и ошибку внешнего сервиса.
5. Убедитесь, что запуск не требует секретов, отсутствующих в документированном файле-примере окружения.
6. Сверьте документацию с фактическими командами и именами файлов.
